Pick your vessel

Same workbench, three different ships. Pack it inside the application's own container, ride a speedboat from your desk, or run a passenger ship the whole team boards.

Container ship

embedded · sidecar · prime use case

Bowire packs into the same container the host app ships in. Add app.MapBowire() and every endpoint is browsable at /bowire, or run the published Docker image as a sidecar pointed at one upstream URL — same effect, different process boundary.

  • Data: per-app, no persistence required
  • Auth: whatever the host pipeline does
  • Crew: as many as the host serves
dotnet add package Kuestenlogik.Bowire
app.MapBowire();

Speedboat

one helmsman · fast · any heading

Installed as a global dotnet tool on the developer's laptop. No fixed route — point Bowire at the staging gRPC service in the morning, a local REST API after lunch, a colleague's WebSocket server in the afternoon.

  • Data: ~/.bowire/ on the local disk
  • Auth: none (you're the only one aboard)
  • Crew: single — you
dotnet tool install -g Kuestenlogik.Bowire.Tool
bowire --url https://staging.example.com

Passenger ship

crew aboard · scheduled routes · per-cabin state

One Bowire server, several authenticated passengers. Operators pin the scheduled port calls; each user keeps their own cabin state — recordings, environments, collections — across sessions. Today: works behind your own OIDC reverse proxy. Coming via the Auth-provider extension SPI: native OIDC + per-user data slots.

  • Data: per-passenger (Phase B)
  • Auth: OIDC plugin (Phase A, opt-in)
  • Crew: as many as the host allows
docker run -p 5080:5080 \
  ghcr.io/kuestenlogik/bowire:latest \
  --url-file /etc/bowire/targets.txt
