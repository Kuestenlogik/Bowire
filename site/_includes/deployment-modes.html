Pick where it runs

The workflows above land on three deployment shapes. Same product, different relationship between Bowire and the rest of your stack.

Embedded · sidecar

prime use case · one app · zero account walls

Add app.MapBowire() to the ASP.NET host and every endpoint is browsable at /bowire. Or run the published container next to the app and point it at one upstream URL — same effect, different process boundary.

  • Data: per-app, no persistence required
  • Auth: whatever the host pipeline does
  • Users: as many as the host serves
dotnet add package Kuestenlogik.Bowire
app.MapBowire();

Local dev tool

one laptop · one user · many target services

Installed as a global dotnet tool on the developer's machine. Point it at whatever you're working on right now — staging gRPC service in the morning, local REST API after lunch, a colleague's WebSocket server in the afternoon.

  • Data: ~/.bowire/ on the local disk
  • Auth: none (you're the only user)
  • Users: single — you
dotnet tool install -g Kuestenlogik.Bowire.Tool
bowire --url https://staging.example.com
preview

Shared instance

team · curated URL space · per-user state

One Bowire server, several authenticated callers. Operators pin a curated set of target URLs; each user keeps their own recordings, environments, and collections across sessions. Today: works behind your own OIDC reverse proxy. Coming via the Auth-provider extension SPI: native OIDC + per-user data slots.

  • Data: per-user (Phase B)
  • Auth: OIDC plugin (Phase A, opt-in)
  • Users: as many as the host allows
docker run -p 5080:5080 \
  ghcr.io/kuestenlogik/bowire:latest \
  --url-file /etc/bowire/targets.txt
