## Next

### Auth-provider extension SPI (Phase A ‚Äî core seam)

Today's `bowire` standalone has no auth surface ‚Äî every endpoint is open on whatever URL the user binds to. That's fine on a laptop, but the moment someone installs Bowire on a shared host the gate has to exist. Rather than ship OIDC (and `Microsoft.Identity.Web`'s transitive weight) in core, give Bowire a third extension type so identity providers slot in the same way protocols and UI widgets do.

- [x] **`IBowireAuthProvider` SPI** ‚Äî sibling to `IBowireProtocol` and `IBowireUiExtension`. Shipped at `src/Kuestenlogik.Bowire/Auth/IBowireAuthProvider.cs` with the two hooks the roadmap called for (`AddAuthentication(IServiceCollection, IConfiguration)` and `BuildDefaultPolicy(AuthorizationPolicyBuilder)`); `BowireAuthProviderRegistry` does the assembly-scan analog to `BowireProtocolRegistry`.
- [x] **`--auth-provider <id>` CLI flag** (default: none) ‚Äî maps to `Bowire:Auth:ProviderId`. When unset, behaviour is identical to today (open workbench). When set, `BowireAuthProviderRegistry.ApplyAuthentication` throws fast with a clear error listing the loaded providers if the named one isn't discoverable.
- [x] **Endpoint integration** ‚Äî `BowireApiEndpoints.Map` builds one route-group (`""`) for every feature-endpoint family and applies `.RequireAuthorization(BowireAuthPolicies.Default)` to the whole group when an `IBowireAuthProvider` is registered. The HTML UI route stays outside the group on purpose (the login flow needs to load the bootstrap HTML before the user can sign in). Embedded mode is unchanged: hosts that don't call `services.AddBowireAuth(builder.Configuration)` keep their own policy.
- [ ] **Plugin-load privilege** ‚Äî auth providers need slightly broader access than protocol plugins (they touch `IApplicationBuilder`, not just `IBowireProtocol`). Either extend `BowirePluginLoadContext` with an `IsAuthProvider` capability flag, or load auth assemblies through a separate load-context. **Deferred** until the first concrete provider (OIDC plugin below) lands and surfaces a real privilege need.

### Auth: OIDC provider plugin (Phase A ‚Äî first concrete impl)

The first concrete auth-provider plugin, riding the SPI above. Ships as `Kuestenlogik.Bowire.Auth.Oidc` (separate NuGet) so the heavy `Microsoft.Identity.Web` dependency only lands in installs that actually use OIDC ‚Äî same MapLibre-style separation that kept ~870 KB of map assets out of the core distribution.

- [x] **`Kuestenlogik.Bowire.Auth.Oidc` plugin** implementing `IBowireAuthProvider` with id `"oidc"`. Shipped at `src/Kuestenlogik.Bowire.Auth.Oidc/`, built on `Microsoft.Identity.Web.AddMicrosoftIdentityWebApi`. Separate NuGet package so `Microsoft.Identity.Web`'s transitive weight only lands in installs that actually need OIDC. Reads its settings from `Bowire:Auth:Oidc:*` (Authority, ClientId, Audience).
- [x] **CLI surface** ‚Äî `--auth-provider oidc` maps to `Bowire:Auth:ProviderId`. Provider-specific keys (`Authority`, `ClientId`, `Audience`) are read straight from configuration ‚Äî operators set them via `Bowire__Auth__Oidc__Authority=‚Ä¶` env vars, appsettings.json, or the standard `--Bowire:Auth:Oidc:Authority ‚Ä¶` form. Dedicated `--auth-oidc-*` shortcut flags are a follow-up once a real install needs them.
- [x] **Single-tenant gate, not multi-tenant**: every authenticated caller still sees the same `~/.bowire/`. Data separation is the explicit Phase-B follow-up below (`IBowireUserStore`).
- [ ] **Required-claim filter** ‚Äî `Bowire:Auth:Oidc:RequiredClaim` (e.g. `groups=bowire-users`) as the access gate ("must be member of group X") on top of the bare RequireAuthenticatedUser policy. Deferred until first install surfaces the need; the current `BuildDefaultPolicy` just requires authenticated.
- [ ] **Token forwarding to target services** ‚Äî auth middleware exposes the access token via `HttpContext.User` so the request pane's Auth tab can pick "Use my session token" as a forwarding mode.

### Multi-tenant data model + SCIM (Phase B ‚Äî blocked on Phase A)

Once Bowire knows who's calling, the next ceiling is "everyone shares one `~/.bowire/`". For real multi-user installs each authenticated identity needs its own slice of state ‚Äî recordings, environments, collections, flows, plugin installs.

- [ ] **User-scoped storage** ‚Äî replace `Path.Combine(homeDir, ".bowire", "environments.json")` with an `IBowireUserStore` that resolves to `~/.bowire-server/users/<sub>/environments.json` (or a real DB). Every consumer (`EnvironmentStore`, `RecordingStore`, `CollectionStore`, `FlowStore`, `PluginManager`) routes through the seam. Standalone single-user mode keeps the flat layout by binding the store to a synthetic "default" user.
- [ ] **SCIM 2.0 endpoints** ‚Äî `/scim/v2/Users` + `/scim/v2/Groups` per RFC 7644. Compliance test suite to verify Okta + Azure AD's provisioning sync round-trips correctly.
- [ ] **Per-user plugin installs** ‚Äî split `~/.bowire/plugins/` into a system-wide tier (admin-managed) plus a per-user overlay so users can install workflow-specific plugins without admin help.
- [ ] **Migration path** ‚Äî single-user installs upgrading into multi-tenant need a one-shot migration that promotes the existing flat `~/.bowire/` into the calling user's slot.

### Plugin lifecycle in the workbench UI + pre-release support

Today's plugin-management surface is asymmetric: the CLI is complete (`bowire plugin list` / `install` / `update` / `update` (all) / `uninstall` / `inspect` / `download`), but the workbench UI only has the install-modal that pops up when a recording references a missing protocol. There's no central place to see what's installed, no update button, no "updates available" badge, no uninstall. Plus the install path strips pre-release versions by default ‚Äî by-design for stable consumers, frustrating when the 1.0-rc cuts of `Bowire.Protocol.Amqp` and `Bowire.Protocol.TacticalApi` need explicit `--version 1.0.0-rc.1` pins to land.

- [ ] **Settings ‚Üí Plugins panel** ‚Äî list every installed plugin with id, version, protocol contributions, "update available" hint (compare to latest on the configured NuGet feed). Per-row action buttons: Update, Uninstall, Inspect. Maps onto the existing `/api/plugins` (list) + new `POST /api/plugins/{id}/update` + `DELETE /api/plugins/{id}` endpoints.
- [ ] **`--prerelease` flag** for `bowire plugin install` / `update` so RC cuts can be pulled without manual version pinning. NuGet's own convention is `--prerelease` on `dotnet add package`; mirroring it keeps the surface predictable. `NuGetPackageInstaller.cs:601` (the `versions.Where(v => !v.IsPrerelease).Max()` line) becomes conditional on the flag.
- [x] **Background update check (opt-in)** ‚Äî `GET /api/plugins/check-updates` queries the latest non-prerelease version of every installed sibling plugin against nuget.org and persists the result to `~/.bowire/state/update-check.json`. UI surfaces a count badge over the Settings gear when pending updates exist; the Plugins panel shows the last-run timestamp and a "Check now" button. **Opt-in** (not opt-out): default off, enable via `--update-check` CLI flag or `Bowire:PluginUpdateCheck:Enabled=true` in appsettings ‚Äî keeps air-gapped / privacy-sensitive installs quiet by default. Implemented in `Plugins/PluginUpdateCheckService.cs` + `Plugins/PluginUpdateCheckHostedService.cs`; documented in [`docs/setup/updating.md`](docs/setup/updating.md#automatic-update-check-opt-in).
- [ ] **Documentation** ‚Äî a short "Updating Bowire and its plugins" page under `docs/setup/`. Covers the tool side (`dotnet tool update`, winget, choco, MSI, standalone) and the plugin side (bundled vs sibling, what comes with the tool, what's separate), so users don't have to read the ROADMAP to find the right command.

### Self-telemetry + Grafana dashboards (Phase B sibling)

Bowire today reports nothing about its own operation ‚Äî fine for a single-user laptop install, but the moment Multi-Tenant Phase B ships, operators of a shared instance need observability the same way they have it for every other service in the stack. The natural backend is whatever the org already runs (Prometheus / Loki / Tempo ‚Üí Grafana); the natural protocol is OpenTelemetry. Optional, opt-in, off by default.

- [ ] **OpenTelemetry integration** ‚Äî `AddOpenTelemetry().WithMetrics().WithTracing()` in the standalone host, opt-in via a `--telemetry` CLI flag and the standard `OTEL_*` env vars. ASP.NET Core's built-in instrumentation covers HTTP-server-side latency / status codes for free; this item is about wiring the registration and gating it on the flag.
- [ ] **Bowire-domain metric set** ‚Äî counters and histograms with stable names so dashboards survive plugin churn: `bowire.invoke.duration` (histogram, tagged by protocol + service + method), `bowire.discover.count` (counter per plugin), `bowire.recording.replay.frames`, `bowire.scan.findings` (counter, tagged by severity + rule-id), `bowire.plugin.load`. Recording bodies / request headers / auth tokens stay off the wire by default; an allow-list of safe metric labels lives in code, not in user config.
- [ ] **Shared `ActivitySource` for embedded mode** ‚Äî when Bowire is mounted into a host app via `AddBowire()`, Bowire's spans should land in the host's existing OTel pipeline, not start a separate one. Use a named `ActivitySource` ("Kuestenlogik.Bowire") that the host can pick up via standard OTel resource attribution.
- [ ] **Reference Grafana dashboard** ‚Äî ship a `dashboards/bowire-overview.json` (Grafana 11+) under `Bowire.Samples` so operators can import a working baseline. Panels: invokes per protocol / hour, p50/p95/p99 invoke latency per plugin, scan-findings trend, plugin-load failure rate, MCP-adapter request rate. Stays positioned as "telemetry about Bowire" ‚Äî Bowire still isn't a dashboarding tool, it's just observable like every other ASP.NET service.
- [ ] **Privacy switch** ‚Äî for shared-host deployments with regulated data, a `--telemetry-strip-method-labels` flag drops the high-cardinality `service.method` dimensions so the metric stream can't be reverse-engineered into "user X called endpoint Y at time T". Standard tradeoff, made explicit.

### Collections (Postman-style test suites)

Complement the existing Recordings feature (auto-captured sessions) and the shipped Flows (visual sequence builder) with manually curated request collections.

- [ ] **Collections** ‚Äî named groups of saved requests, independent of recordings. Each collection item stores: protocol, service, method, body, metadata, expected status ‚Äî everything needed to re-execute the request standalone. Items can be added manually ("Save to collection" from the request pane), imported from a recording, or created from the freeform request builder.
- [ ] **Collection Runner** ‚Äî execute all items in sequence against the active environment. Variable substitution runs per-item, so `${baseUrl}`, `${token}`, etc. resolve fresh. Response chaining between items via `${response.X}` carries values forward.
- [ ] **Per-environment execution** ‚Äî run the same collection against Dev, Staging, Prod by switching the active env. Results stored per (collection, environment) pair so regressions are visible side-by-side.
- [ ] **Persistence + Postman import** ‚Äî JSON files in `~/.bowire/collections/`, synced via the same disk-sync pattern as environments and recordings. Parse Postman Collection v2.1 JSON, map `{{variable}}` ‚Üí `${variable}` automatically.
- [ ] **Flows: Export as test** ‚Äî flatten a flow into a linear collection for CI execution (no visual editor needed in CI).

### Protocol plugins ‚Äî next wave

- [x] **MQTT** (`Kuestenlogik.Bowire.Protocol.Mqtt`) ‚Äî MQTT 3.1.1 / 5.0 via MQTTnet. Topics map to services, publish/subscribe map to unary/streaming. Discovery scans `$SYS/#` or a configured topic prefix. Strong IoT differentiator. **Shipped in core** ‚Äî lives at `src/Kuestenlogik.Bowire.Protocol.Mqtt`, picked up automatically by `BowireProtocolRegistry`.
- **Connect (Buf) support in gRPC plugin** ‚Äî gRPC-compatible RPC with a different wire envelope than gRPC-Web. Extends `BowireGrpcProtocol` rather than spawning a new plugin (proto definitions, descriptors, reflection responses are all reused). _Scope is bigger than gRPC-Web was_: gRPC-Web is a one-line `GrpcWebHandler`-wrap because `Grpc.Net.Client.Web` ships an HTTP handler that translates inside `GrpcChannel`. Connect has no equivalent .NET library (`connect-net` is on the Buf roadmap but not shipped as of 2026-05); the wire envelope, error format, and streaming framing all have to be hand-written. Phased rollout:
  - [x] **Phase 1 ‚Äî unary.** `ConnectInvoker` (hand-rolled, no GrpcChannel): POST `<base>/<service>/<method>`, `Content-Type: application/proto`, `Connect-Protocol-Version: 1` header, single serialised request, 200 + protobuf response on success, 4xx/5xx + JSON `{ code, message }` on error (surfaced as `connect:<code>` status, non-JSON bodies fall back to `http:<status>`). Activated via `connect@<url>` URL hint or `X-Bowire-Grpc-Transport: connect` metadata header (same `GrpcChannelBuilder.ResolveMode` / `ExtractTransportFromUrl` plumbing the Web mode uses). Discovery still routes through gRPC Reflection on HTTP/2 ‚Äî Connect doesn't replace the reflection path in Phase 1.
  - **Phase 1 ‚Äî Unary**. New `GrpcTransportMode.Connect` next to `Native` / `Web`. `connect@<url>` URL prefix + `X-Bowire-Grpc-Transport: connect` metadata header, parallel to the gRPC-Web hooks. Implementation: bypass `GrpcChannel` for Connect mode, drive an `HttpClient` directly ‚Äî POST `/<package>.<service>/<method>` with `Content-Type: application/proto` (or `application/json` for the JSON wire), Connect headers (`Connect-Protocol-Version: 1`, `Connect-Timeout-Ms`), Connect error body parsing (errors are 200 + JSON body, not HTTP status codes). Discovery reuses gRPC reflection where the server exposes it; static-proto-upload path covers servers that don't.
  - [x] **Phase 2 ‚Äî Server-streaming.** `ConnectInvoker.InvokeServerStreamAsync` POSTs one length-prefixed input envelope (`Content-Type: application/connect+proto`, frame = 1-byte flags + 4-byte big-endian length + protobuf payload) and reads the response stream frame-by-frame. Normal frames decode through the descriptor and yield JSON via the existing `IAsyncEnumerable<StreamFrame>` shape; the terminating frame (flag bit `0x02`) carries a JSON `{ error?: { code, message }, metadata? }` envelope ‚Äî if `error` is present, the invoker surfaces a `connect:<code>` trailing frame so the workbench renders it as a stream-end marker instead of silently completing. Wired into `GrpcInvoker.InvokeStreamingWithFramesAsync`: when transport mode is Connect AND the method is server-streaming-only, dispatch hits `ConnectInvoker` instead of falling through to the `GrpcChannel` path (which doesn't speak Connect). Client-streaming + bidi still surface a "Phase 3" not-supported message. 12 new unit tests pin the frame encode/decode round-trip, truncated-header / truncated-payload error paths, the EndStreamFlag bit value, and the end-of-stream JSON parser (with-error, no-error, malformed). Live wire integration against `connectrpc/conformance` waits for Phase 3.
  - **Phase 3 ‚Äî Client- + bidi-streaming**. HTTP/2 with the same envelope framing, full-duplex. Adds the half that gRPC-Web can't carry over HTTP/1.1 ‚Äî Connect-over-HTTP/2 lifts the streaming restriction, but only matters once Phase 1 is in place and someone actually has a Connect-over-HTTP/2 target to hit. Defer to demand.
  - **Tests**: Buf publishes a [`connectrpc/conformance`](https://github.com/connectrpc/conformance) test suite; the reference server is the cheapest target for integration tests.
  - **Marketing surface**: gRPC card on protocols.html mentions Connect alongside native + gRPC-Web; the `connect@` URL prefix shows up in the protocol picker hint.
- [x] **Kafka** (`Kuestenlogik.Bowire.Protocol.Kafka`) ‚Äî Apache Kafka via `Confluent.Kafka` + Schema Registry. Cluster ‚Üî server, topic ‚Üî method, produce ‚Üî unary, consume ‚Üî server-streaming. **Shipped at the sibling repo [`Kuestenlogik.Bowire.Protocol.Kafka`](https://github.com/Kuestenlogik/Bowire.Protocol.Kafka)** ‚Äî latest release [`v1.0.3`](https://github.com/Kuestenlogik/Bowire.Protocol.Kafka/releases/tag/v1.0.3). Single plugin id, two wires picked by URL scheme (`amqp://` / `amqps://` ‚Üí 0.9.1; `amqp1://` / `amqps1://` ‚Üí 1.0). **First stable: [`v1.0.0`](https://github.com/Kuestenlogik/Bowire.Protocol.Amqp/releases/tag/v1.0.0)** ‚Äî first sibling plugin to clear stable. ### AsyncAPI as a discovery source

AsyncAPI is the OpenAPI analogue for event-driven APIs ‚Äî a schema spec that describes channels, operations, messages, and the transport bindings (MQTT, Kafka, AMQP, WebSocket, NATS, ‚Ä¶) those channels use. Rather than ship "AsyncAPI" as its own wire-plugin (it has no wire ‚Äî the wire is whatever the binding says), AsyncAPI lands as a *discovery source* that drives Bowire's existing transport plugins. The mental model matches `bowire --url ./openapi.yaml`: hand Bowire the schema, it builds the method list, calls go out over the right transport.

Built on the official AsyncAPI .NET SDK (`Neuroglia.AsyncApi.Core` + `.IO`, repo: [asyncapi/net-sdk](https://github.com/asyncapi/net-sdk)). The `Client.Bindings.*` packages are *not* used ‚Äî they ship Invocation-side code (own MQTT / Kafka clients etc.) that would duplicate Bowire's existing wire plugins. We use Neuroglia for schema + reading only; wire calls keep going through Bowire's own protocol plugins (MQTT via `Kuestenlogik.Bowire.Protocol.Mqtt`, Kafka via the third-party sibling plugin, ‚Ä¶) at runtime through `BowireProtocolRegistry`.

### Nuclei template compatibility (Bowire.Security.Templates.Nuclei)

The 8000+-template community corpus at [projectdiscovery/nuclei-templates](https://github.com/projectdiscovery/nuclei-templates) is MIT-licensed and the de-facto standard for HTTP web-vulnerability signatures. Reading Nuclei templates natively lifts Bowire's security-scanner from `bwr-vulndb`'s 7 templates to `bwr-vulndb` + nuclei in one stroke ‚Äî and the differentiator stays clean: Nuclei covers HTTP; Bowire's own corpus owns gRPC / GraphQL-deep / WebSocket / OData / MQTT.

- [x] **Phase 2a ‚Äî Project skeleton + reader.** New `src/Kuestenlogik.Bowire.Security.Templates.Nuclei` project consumes `Bowire.Security.Scanner`. `NucleiTemplateReader` walks the YAML representation model via YamlDotNet (same pattern as the AsyncAPI bindings extractor ‚Äî robust against the corpus's actual shape). `NucleiTemplate` POCO covers `id` + `info` + the first `http[]` entry + status/word/regex matchers. `NucleiTemplateConverter.ToBowireRecording` maps metadata + the first HTTP request onto Bowire's recording shape; `VulnerableWhen` predicate is intentionally null until Phase 2b ships matcher translation.
- [x] **Phase 2b ‚Äî Matcher ‚Üí AttackPredicate translation.** `NucleiMatcherTranslator` covers `status` ‚Üí `Status` / `StatusIn`, `word` ‚Üí `BodyContains` (single or AllOf/AnyOf depending on `condition: and/or`), `regex` ‚Üí `BodyMatches` (same compositional shape). `matchers-condition: and/or` composes the per-matcher predicates into `AllOf` / `AnyOf` at the top. `negative: true` wraps in `Not`. `part: body` + `all` + unset are accepted; `part: header` matchers drop out (route through a dedicated translator pass in a follow-up since they ride `HeaderEquals` / `HeaderExists` / `HeaderMissing` rather than the body predicates). Unknown matcher types (`dsl`, `binary`, `size`, ‚Ä¶) drop silently rather than blocking the whole template. Coverage: status/word/regex on body cover the majority of the HTTP web-vuln slice of the corpus.

### Polyglot plugins via sidecar bridge

Bowire plugins today are .NET assemblies implementing `IBowireProtocol`. That locks out teams whose best protocol library lives in Rust (Zenoh), Python (paho-mqtt + the whole IoT/ML stack), Go (NATS core, Temporal), Node.js, or C++. Rather than port every protocol library to .NET, run such plugins as **sidecar processes** and bridge them into the host via JSON-RPC over stdio ‚Äî the same transport LSP, MCP, and DAP settled on.

- [x] **Sidecar plugin contract** ‚Äî full wire spec at [`docs/architecture/sidecar-plugins.md`](docs/architecture/sidecar-plugins.md). JSON-RPC 2.0 with **NDJSON framing** (one envelope per `\n`-terminated UTF-8 line, like MCP's stdio transport ‚Äî no `Content-Length` parsing) maps 1:1 onto `IBowireProtocol`: `initialize` / `shutdown` / `ping` / `discover` / `invoke` / `invokeStream` (host-minted streamId + `$/stream/data` notifications + `$/stream/end`) / `openChannel` + `channel.send` + `channel.close` + `$/channel/data` + `$/channel/closed` (full duplex). Host generates stream / channel ids and subscribes *before* sending the starting request, so a sidecar pushing data immediately can't beat the host to the subscription.
- [x] **Sidecar manifest** ‚Äî `sidecar.json` next to the executable declares `packageId`, `protocol.{id,name,iconSvg}`, `executable`, `args`, `envPrefix`, `shutdownTimeoutMs`, optional `version`. (Named `sidecar.json`, not `plugin.json`, so it never collides with the NuGet-install metadata file the .NET plugin path writes into the shared `~/.bowire/plugins/` tree.) Parser + loader at `src/Kuestenlogik.Bowire/Plugins/Sidecar/SidecarPluginManifest.cs`; defaults applied for optional fields (`envPrefix=BOWIRE_`, `shutdownTimeoutMs=3000`).
- [x] **`SidecarBowireProtocol` adapter** ‚Äî at `src/Kuestenlogik.Bowire/Plugins/Sidecar/SidecarBowireProtocol.cs`. Lazy process spawn (first `DiscoverAsync` / `InvokeAsync` triggers `initialize`; `plugin list` doesn't pay the cost), correlated JSON-RPC request/response, automatic id-mismatch detection on the initialize handshake. `SidecarJsonRpcTransport` owns the process + framing and routes notifications via **per-id subscriptions** (`Subscribe(id)` ‚Üí one `Channel<JsonObject>` per live stream / channel) so concurrent streams + duplex channels never steal each other's frames; graceful shutdown sends `shutdown`, waits `shutdownTimeoutMs`, force-kills on timeout. Duplex channels land via `SidecarChannel : IBowireChannel`. **`SidecarPluginDiscovery`** scans `~/.bowire/plugins/<id>/sidecar.json` files and registers one `SidecarBowireProtocol` per valid manifest; wired into `BowireProtocolRegistry.Discover` so sidecars register alongside .NET plugins with the same `--disable-plugin` handling.
- [x] **Lifecycle + safety (Phase 1)** ‚Äî sidecar crashes surface as JSON-RPC transport errors on the next call; the *following* call respawns (no automatic exponential-backoff loop yet ‚Äî kept simple to debug); per-call timeout via the caller's `CancellationToken`; env-vars matching the manifest's `envPrefix` are forwarded automatically.
- [x] **Channel surface (full duplex)** ‚Äî `openChannel` / `channel.send` / `channel.close` / `$/channel/data` / `$/channel/closed` mapped onto `IBowireChannel` via `SidecarChannel`. Same host-minted-id + subscribe-before-send race-avoidance as `invokeStream`. Sidecar channels report `IsClientStreaming = IsServerStreaming = true` (the protocols that need a channel ‚Äî WebSocket, chat pub/sub ‚Äî are duplex by nature).
- [x] **Dual transport (stdio + HTTP/SSE)** ‚Äî the contract rides over two wires, like MCP's stdio + streamable-HTTP. The manifest's `transport` field picks: `"stdio"` (default ‚Äî spawn `executable`, NDJSON over stdin/stdout) or `"http"` (POST JSON-RPC to `url`, server notifications stream back over one long-lived SSE `GET`). Both implement `ISidecarTransport` and share `SidecarSubscriptionHub` for per-id notification routing; `SidecarBowireProtocol` picks the transport off the manifest. `SidecarHttpTransport` fits hosted / multi-tenant deployments where one (possibly remote) sidecar serves many hosts and Bowire doesn't own the process ‚Äî exactly where the HTTP-callback model beats stdio. Covered by `SidecarHttpTransportIntegrationTests` (discover / invoke / invokeStream-over-SSE / metadata handshake against an in-process fake HTTP+SSE sidecar).
- [x] **End-to-end test coverage** ‚Äî `tests/Kuestenlogik.Bowire.SidecarFake/` ships a tiny .NET console exe that implements the contract; `SidecarBowireProtocolIntegrationTests` spawns it as a real subprocess and exercises `initialize` / `discover` / `invoke` / `invokeStream` / `openChannel`+send+receive round-trips without needing Python / Node / Go on the test host. 11 manifest + discovery unit tests + 5 subprocess integration tests = 16 sidecar tests green; full unit suite stays green.
- [x] **Packaging + install** ‚Äî `bowire plugin install --file <x.zip>` second code path (`PluginManager.InstallSidecarFromZipAsync`): a `.zip` extension (or an `oci://` ref) routes to the sidecar installer (a `.nupkg` keeps the .NET path), accepting a local path, an `http(s)://` URL, or an `oci://registry/repo:tag` reference. The archive must carry a `sidecar.json` at its root (renamed from `plugin.json` to avoid colliding with the NuGet-install metadata file the .NET path writes); packageId is read from the manifest, files unpack into `~/.bowire/plugins/<id>/`, the executable bit is restored on Unix. `bowire plugin list` now tags every entry `[sidecar: <id>]` or `[nuget: N files]`.
- [x] **OCI-registry pull** ‚Äî `bowire plugin install --file oci://ghcr.io/acme/zenoh-sidecar:1.0.0` pulls the sidecar zip straight from any OCI Distribution v2 registry (GHCR, Docker Hub, Harbor, ‚Ä¶). `OciArtifactFetcher` (`src/Kuestenlogik.Bowire.Tool/Plugins/OciArtifactFetcher.cs`) parses the reference (tag or `@sha256:` digest, registry-port-safe), does the anonymous-pull + bearer-token dance (401 ‚Üí `WWW-Authenticate` realm/service ‚Üí token endpoint ‚Üí retry), follows a multi-arch index to its first manifest, picks the zip-ish layer (else first layer), and streams the blob to a temp zip that reuses the existing extract path. 24 parser unit tests (`OciArtifactFetcherTests`) lock down ref / challenge / manifest parsing; the live HTTP path is left to manual / e2e verification.
- [x] **Python SDK** ‚Äî [`Kuestenlogik/Bowire.Sdk.Python`](https://github.com/Kuestenlogik/Bowire.Sdk.Python) (`pip install bowire-plugin`). Subclass `BowirePlugin`, implement `discover` / `invoke` / `invoke_stream` (and optional `settings` / `shutdown`), call `run()` (stdio) **or** `run_http(host, port)` (HTTP/SSE) ‚Äî the same `_Dispatcher` drives both wires, mirroring the host's dual transport.
