# Bowire Roadmap

## In progress

### v1.3.0 — Frame-semantics framework (Phase 1: annotation data model + storage + resolver, Phase 2: built-in detectors + sample-frame probe, Phase 3: MapLibre map viewer + extension framework, Phase 3.1: split-pane layout primitive + selection sync)

ADR: [`docs/architecture/frame-semantics-framework.md`](docs/architecture/frame-semantics-framework.md). Phase 1 lands the C# foundation; Phase 2 lands the auto-detector layer; Phase 3 lands the first concrete viewer + the JS/C# extension surface it rides on. Phase 3.1 lands the layout primitive that lets the map react to multi-selected frames in real time. Phase 4 (right-click UI) and Phase 5 (recording `interpretations` + sample) follow.

#### Phase 3.1 — split-pane layout primitive + selection sync — **shipped**

The map widget needed two refinements before it was actually usable in practice: a way to live ALONGSIDE the streaming-frames list (not just AS another tab), and a way to react to which frames the user has selected. Both ship together because they're entangled UX-wise — selection sync without a visible map next to the list is useless, and split-pane without selection drives no behaviour. No C# changes; the entire surface is in `wwwroot/js`.

- [x] **Split-pane primitive** in a new `wwwroot/js/layout.js`: `createSplitPane(container, { orientation, initialRatio, storageKey })` returns a handle exposing `.firstSlot` / `.secondSlot` HTMLElements (consumers populate; both slots min-clamp at 240px), a draggable divider, a ratio persisted to `localStorage`, and a `dispose()` that detaches the window-resize listener. Mobile fallback (viewport < 720px) stacks vertically regardless of the requested orientation. Divider drag listeners attach to `document` only for the duration of one drag and detach on mouseup — pane recreates across renders never leak handlers. Keyboard nudge via arrow keys (the divider is `tabIndex=0` for accessibility).
- [x] **Per-kind layout defaults + per-(service, method, widget) user override**: `defaultLayoutForKind('coordinate.wgs84')` → `split-horizontal`; every other kind → `tab`. User overrides persist to `localStorage` under `bowire_widget_layout:${serviceId}:${methodId}:${widgetId}` with a forward-compatible shape `{ mode: 'tab' | 'split-horizontal' | 'split-vertical' | 'floating', ratio?: number }`. v1.3.1 ships the `tab` ↔ `split-horizontal` toggle only; `split-vertical` and `floating` are read-tolerant so a future minor version can light them up without a migration step. `floating` (pop-out window) is reserved for v1.x.
- [x] **Layout-toggle UI**: small inline-SVG button in the widget pane header that cycles modes per `cycleLayoutMode`. The same toggle appears in the streaming-frames pane toolbar when the user has switched to `tab` mode (so there's still an escape hatch back to split without going through Phase-4's right-click menu). Non-split-eligible kinds skip the toggle entirely.
- [x] **`selection$` ctx field** (v1.0 ctx additively extends — `frames$`, `selection$`, `theme`, `viewport`, `host`). Async iterable of `{ selectedFrameIds: ReadonlyArray<string|number> }` SNAPSHOTS, not deltas. First `await` yields the current selection so a widget that mounts after the user already made a selection syncs without a manual rehydration step. Implemented as a custom `bowire:frames-selection-changed` DOM event the workbench dispatches exactly once per logical change (N selected frames → one N-entry snapshot, never N delta events). The framework adapts the event stream to async iteration via the existing pattern used for `frames$`.
- [x] **Workbench selection hook** in `render-main.js`: `handleStreamFrameClick(idx, e)` supports plain click (replace), Ctrl/Cmd-click (toggle membership), Shift-click (range from anchor). Visual highlight via a new `.multi-selected` class on each list item; the existing `.selected` class still marks the most recently inspected row (a row can carry both). Multi-select class updates go through `refreshMultiSelectClasses()` — surgical class-toggling rather than re-rendering the list, matching the existing fast-path that keeps the Wireshark-style pane responsive at thousands of frames/sec. Per-frame stable id `${service}/${method}#${index}` minted at the SSE handler in `api.js` (plus the duplex path in `protocols.js`) so snapshots stay portable across method tabs and recordings.
- [x] **Map widget selection logic** in `wwwroot/js/widgets/map.js`: subscribes to `ctx.selection$` inside `mount(...)`. 0 selected → no camera change (preserves user pan/zoom); 1 selected → `flyTo({ center, zoom: 14 })`; N selected → `fitBounds(...)` with ~40px padding. Selected pins render at radius 9 with the theme accent stroke; unselected-but-others-are pins dim to opacity 0.55. Restyle is a single `setData(...)` call with a `selected` property re-flagged per feature; the layer's paint expressions resolve appearance via MapLibre `match` expressions on that property — no layer subtree swap on every Ctrl-click. Stream events still drive pin appending (preserves the 5000-pin cap from Phase 3).
- [x] **Split-pane wired into the streaming output**: `renderStreamingPaneWithWidgets()` decides synchronously whether the active method is split-eligible (built-in extension with `defaultLayoutForKind` of `split-horizontal`) and whether the user has overridden to `tab`. When split is active, the streaming pane and an async-mounted widget pane share the response body via the new primitive. When split is off (or no extension is registered, or no method is selected), falls back to the original `renderStreamingOutput()` unchanged — identical behaviour for every other method.
- [x] **ADR update** in `docs/architecture/frame-semantics-framework.md`: one paragraph in the "v1.0 `ctx` surface" subsection documenting `selection$` (snapshot stream, not deltas; first-await semantics; one event per logical change), and a new "Layout manager" section near "Viewer / editor symmetry" explaining that the layout decision lives in the workbench (not the widget contract), per-kind defaults, per-method user override, persisted to localStorage, and that `floating` is reserved for v1.x.
- [x] **JS bundle concat picks up the new fragment**: `layout.js` slots into the `<BowireJsFragment>` list right before `extensions.js`. Pure vanilla DOM (no `el(...)` dependency) so its position is otherwise unconstrained.

#### Phase 3 — MapLibre bundle + map viewer + extension framework — **shipped**

- [x] **Effective-schema HTTP endpoint** `GET /api/semantics/effective?service={id}&method={id}` in a new `Endpoints/BowireSemanticsEndpoints.cs`. Walks `IAnnotationStore.EnumerateEffective()`, filters by the requested (service, method) pair, and returns `{ service, method, annotations: [{ messageType, jsonPath, semantic, source }] }` — the contract the JS-side extension router consumes when picking widgets to mount.
- [x] **C# extension contract**: new `Kuestenlogik.Bowire.Semantics.Extensions` namespace with `[BowireExtension]` discovery attribute, `IBowireUiExtension` descriptor interface (Id / BowireApiRange / Kinds / Capabilities / BundleResourceName / StylesResourceName), `ExtensionCapabilities` flag enum, and `BowireExtensionRegistry` doing the assembly-scan sweep — same shape as the existing `IBowireProtocol` discovery pattern. The first concrete consumer is `MapLibreExtension` registered against `coordinate.wgs84` with Viewer + Editor capabilities.
- [x] **`/api/ui/extensions` + asset endpoint**: the workbench enumerates discovered UI extensions over JSON (`{ id, bowireApi, kinds, capabilities, bundleUrl, stylesUrl }`) and serves their embedded bundle / stylesheet assets at `/api/ui/extensions/{id}/{name}`. Leaf-filename whitelisting prevents arbitrary file reads from the plugin assembly; the `EmbeddedExtensionAsset` helper tries embedded-resource → dotted-name → on-disk fallbacks so the dev loop never has to rebuild for a fragment-only change.
- [x] **MapLibre GL JS 4.7.1 vendored** under `src/Kuestenlogik.Bowire/wwwroot/maplibre/` (BSD-3-Clause). `maplibre-gl.js` (~800 KB), `maplibre-gl.css` (~64 KB), and the upstream `LICENSE` ship as embedded resources — never a CDN reference, honouring the offline-safe guarantee the ADR pins. The widget bundle injects them via `<script>` + `<link>` from the local Bowire host (`/api/ui/extensions/kuestenlogik.maplibre/{name}`); lazy load so the cost falls only on sessions that actually mount a map.
- [x] **JS extension framework** in `wwwroot/js/extensions.js`: `window.BowireExtensions.register({...})` validates the registration record, applies the `bowireApi: '1.x'` compatibility check, and indexes extensions by `kind`. Mount-decision logic (`mountWidgetsForMethod`) fetches `/api/semantics/effective`, runs pairing-requirement matching grouped by parent JSONPath, mounts every satisfied extension as a tab, and renders a `latitude marked, longitude needed` hint for half-paired annotations. Conflict resolution: built-ins win against same-kind third parties; among same-class extensions, registration order. Phase 4 will hook the user-override picker into this seam.
- [x] **v1.0 `ctx` surface** — frozen for the v1.x lifetime: `frames$` (async iterable backed by the existing SSE pipeline), `theme` ({ mode, accent, font }), `viewport` ({ width, height, on(event, cb) → unsubscribe } backed by `ResizeObserver`), `host` ({ subscribeSse(url), fetch(url, init) }). No recording playback controls, no cross-extension messaging, no first-class persistence helpers — those land additively in v1.1+ if real demand surfaces. `api.js`'s SSE message handler dispatches every parsed frame via a `bowire:stream-message` custom event the framework subscribes to.
- [x] **Map widget** in `wwwroot/js/widgets/map.js`: registers against `coordinate.wgs84` with pairing `[latitude, longitude]` at same-parent scope. The viewer initialises a MapLibre map with a no-source default style (blank background, pins on top — the ADR's "Offline mode" default) when `Bowire:MapTileUrl` is unset, or a raster-tile source when set. Frames coming through `ctx.frames$` get resolved to `(lat, lon)` via the pairing-supplied JSONPaths, projected to pins coloured by `discriminator` value (the ADR's "one layer per discriminator" sketch), and trimmed to 5000 points to bound memory on long streams. The v1.3-minimum editor implementation lets the user click to place a draggable marker that calls `ctx.onChange({ latitude, longitude })` on every move — full Phase-4 drag-and-edit surface is deferred.
- [x] **JS bundle concat picks the new fragments up**: `extensions.js` is wired into the `<BowireJsFragment>` list BEFORE `widgets/map.js` so `window.BowireExtensions.register` exists when the widget calls it; `init.js` stays last. The `Watch` glob widens to `wwwroot/js/**/*.js` to cover the new `widgets/` subdirectory.
- [x] **Tests**: new `tests/Kuestenlogik.Bowire.Tests/Semantics/Endpoints/MapLibreExtensionTests.cs` covers the descriptor + the registry-discovery sweep. `tests/Kuestenlogik.Bowire.IntegrationTests/BowireSemanticsEndpointsTests.cs` boots an isolated `WebApplication` with `AddBowire(opts => opts.SchemaHintsPath = "")`, seeds the auto-detector layer, and walks the effective-schema response shape end-to-end — verifies the `source` field tracks `User > Plugin > Auto`, the `(service, method)` filter rejects unrelated annotations, BadRequest fires on missing query params, and the asset endpoint serves the map bundle + MapLibre LICENSE while rejecting non-whitelisted paths. `SemanticsEndpointIntegrationTests` exercises the same surface against the shared `BowireTestFixture`.

#### Phase 1 — Annotation data model + storage + resolver — **shipped**

- [x] **Core types** under a new `Kuestenlogik.Bowire.Semantics` namespace: `AnnotationKey` (four-dimensional `service / method / message-type / json-path` record with `"*"` wildcard for single-type methods), `SemanticTag` (strongly-typed kind-string wrapper with `BuiltInSemanticTags` catalogue covering the ADR's well-known kinds — coordinate.{latitude,longitude,ecef.*}, image.bytes, audio.bytes, timeseries.*, table.row-array, plus the explicit-suppression `none`), `Discriminator` (kind ∈ {wirePath, jsonPath, oneof, none}) and `Annotation` (key + tag + source).
- [x] **Storage layer**: `IAnnotationStore` read-surface with built-in resolution priority; `InMemoryAnnotationLayer` concurrent dictionary backing the session-scope edits; `JsonFileAnnotationLayer` for the user-local + project-local files (`bowire.schema-hints.json` — byte-for-byte the format the ADR pins under "Persistence", async load + save, write-to-temp + atomic-rename so concurrent writers never tear the file).
- [x] **`LayeredAnnotationStore`** composes the four layers (session in-memory, project file, user file, auto-detector in-memory) plus the plugin-hints callback, enforcing `User > Plugin > Auto` resolution priority — `SemanticTag.None` from a higher tier suppresses lower tiers, exactly as the ADR specifies.
- [x] **`IBowireSchemaHints`** optional companion interface to `IBowireProtocol`. A plugin that knows the shape of its payloads in advance (DIS, TacticalAPI) implements both on the same class; the protocol-registry pulls hints at resolver-call time. Returning empty is the default and is exactly what content-agnostic plugins (REST, GraphQL, generic gRPC) should do.
- [x] **DI registration**: `BowireServiceCollectionExtensions.AddBowire(configure)` overload wires `LayeredAnnotationStore` as a singleton (also exposed as `IAnnotationStore`). User-local layer defaults to `~/.bowire/schema-hints.json`; override via the new `BowireOptions.SchemaHintsPath` (empty string disables the user file entirely for hardened deployments). Project-local layer is auto-discovered as `bowire.schema-hints.json` in the current working directory, opted out by absence.
- [x] **Tests**: 59 new unit tests in `tests/Kuestenlogik.Bowire.Tests/Semantics/`. Resolution-priority matrix is exhaustive (every present/absent × User/Plugin/Auto combination + `None`-suppression at every tier + the session/project/user within-User-tier ordering). JSON round-trip test validates both directions against the ADR's literal DIS + Harbor example. File-locking test runs 32 interleaved `SaveAsync` calls from two `JsonFileAnnotationLayer` instances against the same path and asserts the final file is well-formed and contains exactly one of the two starting states.

#### Phase 2 — built-in detectors + sample-frame probe — **shipped**

The auto-detector half of the framework: five stateless `IBowireFieldDetector` implementations + a thread-safe `FrameProber` that walks each `(service, method, message-type)` triple exactly once (ADR's "learn-as-new-types" property, step 7 of "Lifecycle of a binding") and writes proposals into the existing `LayeredAnnotationStore.AutoDetectorLayer`. No UI work, no recording changes — those are still Phase 3 and Phase 5.

- [x] **Detector contract** under a new `Kuestenlogik.Bowire.Semantics.Detectors` subnamespace: `IBowireFieldDetector` (`Id` + `Detect(in DetectionContext)`), `DetectionContext` (`readonly record struct` carrying the four-dimensional addressing context + the decoded `JsonElement` root), `DetectionResult` (`readonly record struct` of `(AnnotationKey, SemanticTag)`). Detectors receive the whole decoded frame so paired-field rules (lat+lon at same parent) match naturally. Shared low-level helpers — base64-prefix decode, magic-byte match, depth-first JSON walk with JSONPath construction — live in an internal `DetectorHelpers` class so each detector stays focused on its own rule.
- [x] **Five built-in detectors**, all stateless / thread-safe, all opt-outable as a set via `BowireOptions.DisableBuiltInDetectors`:
    - `Wgs84CoordinateDetector` — paired `lat(itude)?` + `lo?ng(itude)?` fields at the same parent path, values in `[-90, 90]` and `[-180, 180]` respectively. **Strict v1**: both names must match — range-only matching is disabled so a `{x: 5.2, y: 9.1}` pixel-coord pair doesn't drag the map widget onto a non-geographic field pair. Ambiguous shapes (two lat-shaped names at the same parent) deliberately don't fire either.
    - `GeoJsonPointDetector` — objects with `type == "Point"` and a 2- or 3-element `coordinates` array of numbers. Emits `coordinate.longitude` on `coordinates[0]` and `coordinate.latitude` on `coordinates[1]` (GeoJSON's lon-first ordering). The altitude slot in 3D points is left unannotated (no built-in `coordinate.altitude` kind in v1).
    - `ImageBytesDetector` — string-typed leaves whose first 12 base64-decoded bytes match PNG (`89 50 4E 47 0D 0A 1A 0A`), JPEG (`FF D8 FF`), GIF87a / GIF89a, WebP (RIFF + WEBP at offset 8), or AVIF (ftyp + avif brand at offset 4). Only the 12-byte prefix is decoded — bounded work regardless of payload size.
    - `AudioBytesDetector` — same shape, magic table covers WAV (RIFF + WAVE at offset 8), Ogg (`OggS`), FLAC (`fLaC`), MP3 frame sync (`FF FB` / `FF F3` / `FF F2`), and ID3v2 tags.
    - `TimestampDetector` — `*timestamp*` / `*time*` / `*at$` name pattern combined with an ISO-8601 string (`DateTimeOffset.TryParse` with `DateTimeStyles.RoundtripKind`) or a number in the ±20-year plausibility window around "now" (seconds- or millisecond-epoch). `TimeProvider` test-seam in the constructor so unit tests pin the window to a deterministic clock instead of depending on real time. The implausible-tiny-epoch case (`42 seconds since 1970`) and the "at"-substring-but-not-suffix case (`data`, `atmosphere`) are both explicitly rejected.
- [x] **`IFrameProber` + `FrameProber`** — singleton-scoped composer of the detector set against `LayeredAnnotationStore.AutoDetectorLayer`. Tracks probed `(serviceId, methodId, messageType)` triples in a `ConcurrentDictionary<(string, string, string), byte>`; `ObserveFrame` is a single dictionary lookup for the repeat case (the hot path on a streaming SSE channel), runs every detector exactly once for novel triples. A misbehaving detector that throws on a specific frame shape doesn't take the stream down — the prober swallows the exception and continues with the next detector. Step 7 of the ADR's "Lifecycle of a binding" — learn-as-new-types — is preserved by construction.
- [x] **Stream hook** in `BowireInvokeEndpoints.MapBowireInvokeEndpoints` (the existing `/api/invoke/stream` SSE handler). One added line per frame on each of the two existing emit-paths (`IBowireStreamingWithWireBytes` path + the plain `InvokeStreamAsync` path) calls `FrameProbingMiddleware.Observe(...)` — a tiny static helper that parses the frame JSON, builds a `DetectionContext` with `AnnotationKey.Wildcard` as the message-type (Phase-2 limitation — plugin-side discriminator wiring lands in a later phase), and hands it to the prober. The middleware swallows malformed JSON / null frames silently because the prober is an annotation side-channel, not a stream-breaking concern.
- [x] **DI registration**: the five detectors registered as `IEnumerable<IBowireFieldDetector>` plus as their concrete types (for diagnostics); `IFrameProber` registered as singleton wired to the existing `LayeredAnnotationStore`. New `BowireOptions.DisableBuiltInDetectors` opt-out skips the five-detector registration but still wires the prober — hardened deployments can layer their own detector set on top.
- [x] **Tests**: 81 new unit tests in `tests/Kuestenlogik.Bowire.Tests/Semantics/Detectors/`, one fixture per detector + a prober fixture + a DI-surface fixture. Every detector has its false-positive shape covered explicitly — `{x: 5.2, y: 9.1}` mustn't match WGS84; `pixelX/Y` in the right range mustn't match either; WebP magic mustn't trigger the audio detector; an `atmosphere` field mustn't trigger the timestamp detector; a tiny `42` mustn't be read as an epoch second. The prober test covers same-triple-observed-twice (one detector call), different-triple (fresh probe), and throwing-detector-doesn't-break-the-rest. Total unit + mock + integration suite stays green: 1403 / 409 / 163.

### gRPC-Web transport opt-in for the gRPC plugin (2026-05-11)
- [x] **gRPC plugin gains a gRPC-Web transport variant.** Same plugin, same `.proto`, same service catalogue — opt-in flips the inner pipeline to wrap `SocketsHttpHandler` in `GrpcWebHandler`, pins the HTTP version to 1.1, and routes gRPC framing through the gRPC-Web encoding so callers can hit browser-fronted services and HTTP/1.1-only ingresses (Envoy / grpcwebproxy / ASP.NET `UseGrpcWeb`). Typical case: services like Rheinmetall's TacticalAPI that expose both native gRPC on `:4267` and gRPC-Web on `:4268`.
- [x] **Two opt-in paths**, in priority order:
    1. **URL hint** — `grpcweb@http://host:4268` resolves to the gRPC plugin with web mode, symmetric to the existing `grpc@…` hint.
    2. **Metadata header** — `X-Bowire-Grpc-Transport: web` (exposed as `BowireGrpcProtocol.TransportMetadataKey`); any other value (or absence) falls back to native HTTP/2.
- [x] **Plain `grpc@` and bare URLs stay on native HTTP/2** unchanged — all existing tests stay green.
- [x] **mTLS composes with web mode**: when both an mTLS marker and the web transport hint are present, `MtlsHandlerOwner`'s `SocketsHttpHandler` becomes the inner of the `GrpcWebHandler`.
- [x] **Internal helper extraction**: a single `GrpcChannelBuilder.BuildChannel(serverUrl, innerHandler, mode)` is the only place that constructs a `GrpcChannel` from a handler — the reflection client, the invoker, and the duplex channel all route through it.
- [x] **URL-hint surface extension**: `BowireEndpointHelpers.ResolveHint` is the new precedent for "hint maps to plugin id + side-channel metadata". `grpcweb` → `("grpc", X-Bowire-Grpc-Transport=web)`. Future transport-mode hints (Akka classic/cluster, MQTT v3/v5, ...) plug into the same table. Discovery — which has no metadata bag on the public interface — piggy-backs the mode on a `__bowireGrpcTransport=web` query marker that the plugin strips before opening the channel.
- [x] **Known limitation**: **duplex / client-streaming over `OpenChannelAsync` returns `null` in web mode.** gRPC-Web's HTTP/1.1 framing can't carry duplex traffic; even `GrpcWebMode.GrpcWebText`'s client-streaming round-trip behaves inconsistently across server implementations (Envoy, grpcwebproxy, ASP.NET disagree on flush semantics). Unary + server-streaming work cleanly through `InvokeAsync` / `InvokeStreamAsync`. Future track: revisit if upstream stabilises duplex over gRPC-Web's WebSocket extension.

### v1.1.0 — Standalone-CLI URL, coverage hardening, MCP docs (2026-05-11)
- [x] ⚠ **Standalone CLI now mounts the workbench at `/` instead of `/bowire`.** The standalone `bowire` tool has no host app to share routes with, so the `/bowire` prefix was a wasted hop. New URLs: workbench at `http://localhost:5080/`, MCP adapter at `http://localhost:5080/mcp`. Embedded callers are untouched — `app.MapBowire()` (no pattern arg) keeps defaulting to `/bowire`; `app.MapBowire("/your/prefix")` keeps mounting wherever you told it to. The `/` → `/bowire` redirect shim is gone (no longer needed). Mechanics: `BowireApiEndpoints.Map` pre-computes `basePath` once and passes it to every sub-endpoint registration; `BowireHtmlGenerator` mirrors the same logic for the JS-side `config.prefix`; `WithMcpAdapter` does the same normalisation.
- [x] **gRPC reflection warning on standalone start removed**. Two latent bugs combined to log `"Protocol plugin grpc failed to map discovery endpoints"` on every standalone startup: (1) `BrowserUiHost` never called `builder.Services.AddBowire()`, so plugins' `IBowireProtocolServices.ConfigureServices(...)` was skipped; (2) `BowireApiEndpoints.Map` ran every plugin's `MapDiscoveryEndpoints(...)` unconditionally, but those endpoints only make sense in **Embedded** mode where Bowire is mounted inside a host that's actually a gRPC / SignalR server. Standalone CLI is a client — there's no host surface to reflect. Fix: AddBowire() now runs in the standalone host, and `MapDiscoveryEndpoints(...)` is gated on `options.Mode == BowireMode.Embedded`.

### Systemic Re-render Fix via morphdom (v0.9.2)
- [x] **Root cause of the recurring re-render pain**: `render()` was doing `app.innerHTML = ''; app.appendChild(renderSidebar()); app.appendChild(renderMain());` on every state change. Any UI region that needed live updates during streaming, typing, or scrolling paid the cost of a full sidebar + main rebuild per tick — burnt focus, dropped scroll position, O(sidebar + history) work per message.
- [x] **morphdom 2.7.4 vendored** as `wwwroot/js/_morphdom.js` (~12 kB UMD minified). First entry in the `BowireJsFragment` list so it lives **outside** the IIFE opened by `prologue.js` — the UMD wrapper sets `window.morphdom` on the global scope, and the rest of the Bowire code inside the closure references it via `window.morphdom(...)`.
- [x] **`render()` rewritten** (`render-env-auth.js`): builds the next DOM tree in an off-screen `<div id="bowire-app">`, then calls `window.morphdom(app, next, { childrenOnly: true, onBeforeElUpdated })` to diff/merge into the live tree. Preserves focus, selection, scroll position, and existing event listeners wherever the old and new trees agree structurally.
- [x] **Active-input protection** in `onBeforeElUpdated`: when the currently focused element is an `INPUT` / `TEXTAREA` / `SELECT`, copies `fromEl.value` onto `toEl.value` *before* morphdom's special handler runs (which would otherwise overwrite it with the rebuilt value), and restores `selectionStart`/`selectionEnd` on the next frame so the cursor doesn't jump mid-typing.
- [x] **isEqualNode fast-path**: structurally identical subtrees (static toolbars / headers that haven't changed) skip the whole morph entirely. Big perf win for the 80% of the UI that's stable across renders.
- [x] **Graceful fallback** if morphdom fails to load for any reason (stale cache, EmbeddedResource build error): falls back to the old `replaceChildren` + rebuild path so the UI stays functional.
- [x] **Why not Lit** (revisiting the v0.9.1 roadmap proposal): morphdom solves the systemic re-render problem with a **one-function patch** against the existing vanilla JS architecture. Lit would have required rewriting every `el()`-based render path into `html\`\`` templates, introducing Shadow DOM style-isolation concerns, and paying for component encapsulation features we don't actually need. morphdom is the right-sized tool for the problem Bowire actually has. The Streaming UI append-fast-path from v0.9.1 is now technically redundant — a plain `render()` would be cheap enough with morphdom — but we're keeping it because it's already written, battle-tested, and gives us sub-millisecond latency on hot-path streaming messages.

### Polyglot plugins via sidecar bridge

Bowire plugins today are .NET assemblies implementing `IBowireProtocol`. That locks out teams whose best protocol library lives in Rust (Zenoh), Python (paho-mqtt + the whole IoT/ML stack), Go (NATS core, Temporal), Node.js (gRPC-Web tooling), or C++ (legacy middleware). Rather than port every protocol library to .NET, Bowire could run such plugins as **sidecar processes** and bridge them into the host via JSON-RPC over stdio — the same transport LSP, MCP, and DAP settled on for exactly this polyglot problem.

- [ ] **Sidecar plugin contract** — a small JSON-RPC 2.0 protocol over stdio (newline-delimited or Content-Length-framed, TBD) that maps 1:1 onto `IBowireProtocol`:
    - `bowire.discover(serverUrl, showInternalServices)` → returns the service/method tree as JSON
    - `bowire.invoke(serverUrl, service, method, messages, metadata)` → returns unary `InvokeResult`
    - `bowire.invokeStream(...)` → notifications stream `bowire.frame`/`bowire.done`/`bowire.error` back until the iterator ends
    - `bowire.openChannel(...)` / `bowire.channel.send` / `bowire.channel.close` → duplex channels, similar notification stream
    - Lifecycle: `initialize` / `shutdown` / `ping` (`exit` as last-resort kill)

- [ ] **Sidecar manifest** — each sidecar plugin ships a `plugin.json` next to its executable:
    ```json
    {
      "packageId": "acme-zenoh-plugin",
      "protocol": { "id": "zenoh", "name": "Zenoh", "iconSvg": "..." },
      "executable": "zenoh-bridge",
      "args": ["--stdio"],
      "envPrefix": "ACME_ZENOH_"
    }
    ```
    The host spawns the executable, pipes JSON-RPC over stdin/stdout, captures stderr for diagnostics.

- [ ] **`SidecarBowireProtocol` adapter** in the core — implements `IBowireProtocol` by translating every method call into a JSON-RPC request over stdio. One adapter instance per sidecar process. Streaming maps to JSON-RPC notifications on the same pipe (MCP already does this, so the encoding is well-trodden).

- [ ] **Per-language SDKs** — minimal helper libraries that give plugin authors a typed server shell instead of reimplementing JSON-RPC framing:
    - **Python**: `pip install bowire-plugin`, subclass `BowirePlugin` with `discover` / `invoke` / `invoke_stream` methods, run via `python -m bowire_plugin`.
    - **Node.js / TypeScript**: `npm i @bowire/plugin`, export a plugin object, entry point drains stdio.
    - **Go**: `bowire/go/plugin` package exposes a `Serve(ctx, plugin)` function that runs the stdio loop.
    - **Rust**: `bowire-plugin` crate with `async` trait.

- [ ] **Lifecycle + safety** — sidecar crashes surface as protocol errors rather than taking down the host; auto-restart on unexpected exit (exponential backoff); configurable timeout per call; inherit `BOWIRE_PLUGIN_DIR` / target URLs via env so sidecars don't need their own CLI-arg story. The existing `BowirePluginLoadContext` isolation story doesn't apply (sidecars are separate OS processes, already isolated) — the threat model shifts from "version-clash prevention" to "process-isolation + resource limits".

- [ ] **Packaging + install** — `bowire plugin install` needs a second code path for sidecar plugins: instead of pulling a NuGet package, it fetches a zipped release artifact (from GitHub Releases, a private feed, or an OCI artifact registry) and unpacks the executable + `plugin.json` into `~/.bowire/plugins/<id>/`. Platform-specific binaries (Windows/Linux/macOS × x64/arm64) picked by the `bowire` runtime. Mixed plugin collections (some NuGet, some sidecar) all appear in `bowire plugin list` with a `kind: nuget | sidecar` marker.

- [ ] **Template** — `dotnet new bowire-plugin --sidecar python` scaffolds a Python plugin with the right `plugin.json`, a Dockerfile/release-zip workflow, and a `Sample` protocol implementation. Separate NuGet Templates repo ([Kuestenlogik/Bowire.Templates](https://github.com/Kuestenlogik/Bowire.Templates)) — the `--sidecar <lang>` parameter picks the language.

Unlocks: Rust-based Zenoh plugin, Python plugins for every paho-mqtt / aiokafka / AWS-IoT library, Node.js Socket.IO bridge that uses the real socket.io-client instead of hand-rolling the protocol, Go plugins for Temporal / NATS JetStream. Larger architectural lift than any single new protocol plugin, but lifts the ceiling on what Bowire can ever cover.

### Collections + Flows (Postman-style test suites)

Complement the existing Recordings feature (auto-captured sessions) with manually curated request collections and conditional execution flows.

- [ ] **Collections** — named groups of saved requests, independent of recordings:
    - Each collection item stores: protocol, service, method, body, metadata, expected status — everything needed to re-execute the request standalone.
    - Items can be added manually ("Save to collection" from the request pane), imported from a recording, or created from the freeform request builder.
    - **Collection Runner** — execute all items in sequence against the active environment. Variable substitution runs per-item, so `${baseUrl}`, `${token}`, etc. resolve fresh. Response chaining between items via `${response.X}` carries values forward.
    - **Per-environment execution** — run the same collection against Dev, Staging, Prod by switching the active env. Results are stored per (collection, environment) pair so regressions across environments are visible side-by-side.
    - **Persistence** — JSON files in `~/.bowire/collections/`, synced via the same disk-sync pattern as environments and recordings. Importable/exportable, shareable via workspace files.
    - **Postman import** — parse Postman Collection v2.1 JSON into Bowire collections. Map Postman's `{{variable}}` syntax to Bowire's `${variable}` automatically.

- [x] **Flows** — visual sequence builder for conditional, multi-step API interactions:
    - ~~DAG-style editor (not just linear)~~: ✅ Nested branching model — condition nodes have inline trueBranch/falseBranch child arrays, loop nodes have body arrays. Vertical pipeline with indented branches.
    - ✅ **5 node types**: Request (REQ), Condition (IF), Delay (WAIT), Variable (VAR), Loop (LOOP).
    - ✅ **Condition branching** — IF nodes split into true/false paths rendered side-by-side with merge indicators. 6 operators: ==, !=, >, <, contains, exists.
    - ✅ **Loop nodes** — count-based (N iterations) and while-condition (repeat until condition fails). Nested body with add-node zone.
    - ✅ Variable forwarding between nodes via `${response.X}` — the output of one node feeds the input of the next.
    - ✅ **Recursive runner** executes the node tree top-to-bottom, following condition branches and looping body arrays, with live pulse animation on the active node.
    - ✅ **Response viewer** — expandable per-node result panel showing response JSON, error messages, and timing.
    - ✅ **Import/Export** — flows as `.bwf` JSON files. Duplicate flow, import from file picker.
    - ✅ **Sidebar tab** (4th tab) with flow list, create/select. Visual node editor in main pane.
    - ✅ **Drag-to-reorder**, inline editing (click-to-expand), move up/down, duplicate, delete per node.
    - [ ] **Export as test** — flatten a flow into a linear collection for CI execution (no visual editor needed in CI).
    - Natural follow-on from the freeform request builder: flows connect freeform requests into automated scenarios.

Both features build on the existing `IBowireProtocol.InvokeAsync` path — collections and flows are orchestration layers, not new protocol work. The Collection Runner is essentially a generalized version of `replayRecording()` with environment awareness and response chaining. Flows add branching and looping on top.

### Replay-Mock-Server — recordings as API endpoints

Turn any Bowire recording into a standalone mock server that replays the captured responses — no live backend required. Competitive landscape: Postman's mock servers are cloud-hosted, WireMock is the Java gold standard but heavyweight, Bruno has community plugins, Insomnia outsources to third-party tools. Bowire's angle: local-first, multi-protocol, re-uses the existing recording format end-to-end — no separate mock-config DSL, no cloud account.

**Two orthogonal trigger modes** the mock has to support:

- **Reactive** — request comes in, response goes out. Unary REST, unary gRPC, but also streaming endpoints that are triggered by a client connect (SSE, gRPC server-streaming, WebSocket upgrade, SignalR hub method) — the client initiates, the mock replays the captured response sequence at the original timing.
- **Proactive** — no inbound request. A clock drives the emission: MQTT publishes on a schedule onto the recorded topics, DIS broadcasts the captured PDU sequence, a test SSE endpoint pushes every N seconds regardless of subscribers. The mock acts as a source, not a responder.

Both modes share the same recording format and the same per-frame timestamp infrastructure; they differ only in what starts the emission (client connect vs. timer tick).

Shipped in three numbered phases so the MVP can land first and later phases are independent, optional add-ons.

#### Phase 1a — Static Replay-Mock (MVP), REST unary only &mdash; **shipped**

The minimum that unblocks the main use case: "I want to test my frontend / service against a reproducible API without running the real backend." Scope was deliberately narrow — **reactive + unary, REST only**. gRPC unary moves to Phase 1b (needs proto-descriptor work); streaming, timed emission, and multi-protocol arrive in Phase 2 once the recording-format groundwork is in.

#### Phase 1b — gRPC-unary replay &mdash; **shipped**

Adds gRPC unary replay on top of the Phase-1a package. The non-obvious part was that gRPC responses are binary-framed protobuf with trailers and `Google.Protobuf`'s C# library has no `DynamicMessage` equivalent — a runtime JSON→binary encoder built from only a proto descriptor isn't possible without codegen. The pragmatic fix: **capture the binary wire bytes at recording time**. The mock emits them 1:1 at replay time; no runtime re-encoding, no `.proto` files needed alongside the recording.

- [x] **Recording-format v2** — `recordingFormatVersion: 2` bumped on every new recording. gRPC unary steps now also carry `responseBinary` (base64 of the protobuf wire bytes). v2 is a strict superset of v1: steps without `responseBinary` replay as v1 semantics (REST-only).
- [x] **Binary capture on the recorder side** — `InvokeResult.ResponseBinary` (byte[]?) flows through the gRPC plugin's `GrpcInvoker` (unary + client-streaming paths), into the invoke-endpoint response as base64, and into the client-side `captureRecordingStep(...)` call.
- [x] **Dynamic gRPC server binding** — `UnaryReplayer` writes the recorded bytes with correct gRPC framing: HTTP/2 response, `Content-Type: application/grpc`, 5-byte length-prefix + payload, `grpc-status` (+ optional `grpc-message`) trailer. Status-code mapping covers the full 0..16 range per the grpc.io spec.
- [x] **Matcher extension** — `ExactMatcher` gained a gRPC code path: matches `application/grpc`-content-type requests by the `/{service}/{method}` URL path (package dots treated as part of the service name, not path separators). REST requests continue to match by `(verb, path)`.
- [x] **Listener protocol auto-selection** — the standalone `bowire mock` listener picks HTTP/2 prior-knowledge when the recording contains any gRPC step, else HTTP/1.1. Plaintext HTTP/1+HTTP/2 can't share a port without TLS+ALPN; documenting the choice rather than forcing TLS keeps the UX simple.
- [x] **Tests + docs** — 3 new gRPC-specific tests in `Kuestenlogik.Bowire.Mock.Tests` (round-trip with `Grpc.Net.Client` + `StringValue`, unknown-method rejection, status-to-grpc-code mapping), plus 5 new matcher tests. Docs page updated with the new "gRPC replay" and "Protocol selection" sections.
- [x] **Phase 1c** &mdash; **shipped**: embed the proto `FileDescriptorSet` in the recording (optional `schemaDescriptor` field on gRPC steps, populated at discovery time from gRPC Server Reflection) and expose gRPC Server Reflection on the mock via `Grpc.Reflection.ReflectionServiceImpl`. Descriptor pool is built once at startup across every unique `schemaDescriptor` in the recording; duplicate and transitive dependency files are de-duplicated by name; topo-sorted by proto dependency order. Pipeline structure: `UseRouting` → mock middleware (passes through when a specific endpoint matched) → `UseEndpoints` (reflection service) → `app.Run` 404 fallback for truly unmatched paths. Two new integration tests (`FileContainingSymbol`, `ListServices`) drive a real `Grpc.Reflection.V1Alpha.ServerReflection.ServerReflectionClient` against a running `MockServer`; both round-trip through the reflection protocol via HTTP/2 prior-knowledge. With this, a second Bowire workbench pointed at the mock auto-discovers every mocked service — no `.proto` file needs to be shared out of band.

#### Phase 2 — Streaming, timed emission, dynamic values, multi-protocol

Once Phase 1 has landed and the per-frame timestamps are flowing into the recordings, Phase 2 adds the streaming and timed-emission dimensions Postman / WireMock routinely offer — plus the remaining protocols.

- [x] **Reactive + streaming replay — SSE** &mdash; **Phase 2c, shipped.** When a client hits a recorded REST server-streaming step, the mock writes a `text/event-stream` response and emits each captured frame as a `data: <payload>\n\n` event, paced by the per-frame `timestampMs` that the recorder stored. The outer envelope the recorder used for the Bowire UI (`{index, data, timestampMs}`) is stripped from the wire — clients see just the inner `data` payload, matching what the original backend would have produced. Dynamic-value substitution (`${uuid}`, `${now}`, …) runs on each frame individually, so streamed traffic carries fresh per-request values. `MockOptions.ReplaySpeed` controls pacing (1.0 default, 0 = instant, 2.0 = 2× speed); a `--speed` CLI flag is a follow-up polish item. Recorder-side: extended api.js's streaming capture to persist the full `receivedMessages` array (index + timestampMs + data) instead of just the last frame. Steps captured by pre-2c builds get a clear `501 — receivedMessages missing, re-record` rather than an empty stream.
  - [x] **gRPC server-streaming replay** &mdash; **Phase 2d, shipped.** New optional `IBowireStreamingWithWireBytes` interface (implemented by `BowireGrpcProtocol`) yields `StreamFrame(Json, Binary)` per frame. The invoke-endpoint detects the interface and records `responseBinary` per SSE frame; the mock replays with the correct 5-byte length-prefix HTTP/2 framing and per-frame `timestampMs` pacing, closing with the mapped `grpc-status` trailer. End-to-end tested with `Grpc.Net.Client`'s `AsyncServerStreamingCall` + `StringValue`.
  - [x] **gRPC client-streaming replay** &mdash; `ReplayGrpcClientStreamAsync` accepts the incoming HTTP/2 stream, drains every 5-byte-prefixed request message the client sends, then writes the single recorded `responseBinary` frame plus the mapped `grpc-status` trailer. Request-frame content is deliberately not compared against `sentMessages` — order-based validation for binary protobuf would need schema-aware decoding the mock intentionally avoids, so recordings with out-of-order request shapes still replay cleanly. Verified against `Grpc.Net.Client`'s `AsyncClientStreamingCall` + three-frame `StringValue` upload.
  - [x] **gRPC bidirectional-streaming replay** &mdash; `ReplayGrpcBidiAsync` merges the step's `sentMessages` + `receivedMessages` into a single wall-clock timeline (same pattern as WebSocket input-gating) and walks it: received frames emit paced by `timestampMs`, sent frames act as gates that block until the client has written a corresponding request frame. Order-based matching, no content inspection. Recordings without `sentMessages` degrade to pure server-push so a pre-gating bidi capture still replays. Verified against `Grpc.Net.Client`'s `AsyncDuplexStreamingCall` with a three-recv / two-gate timeline.
  - [x] **WebSocket replay** &mdash; **Phase 2e, shipped.** When a client opens `ws://<mock>/<path>`, the mock accepts the upgrade and pushes the captured `receivedMessages` back paced by `timestampMs`, using `WebSocketMessageType.Text` for `{type:"text",...}` envelopes and `.Binary` for `{type:"binary",base64:"..."}`. Recorder-side: duplex-channel capture in `protocols.js` now carries `httpPath`+`httpVerb`; the WebSocket plugin's discovery sets `HttpMethod = "GET"` on every method info so the standard REST matcher pairs incoming upgrade requests with recorded steps. MockServer pipeline grew an explicit `UseWebSockets()` step.
    - **Input-gating (Phase 2e+):** when the recording also carries `sentMessages`, those act as synchronization points on a merged wall-clock timeline — the replayer emits received frames up to the next sent checkpoint and blocks until the client transmits a corresponding frame, so request/response duplex flows stay in lockstep. Order-based matching (the N-th client frame advances past `sentMessages[N]`, content not inspected). Recordings without `sentMessages` keep the original fire-and-drain behaviour so pure server-push captures stay unaffected. Ticket-based signalling via `Channel<byte>` between the receive loop and the replay loop; virtual time advances to the gate timestamp on release so the next received frame paces from the gate-unlock moment rather than the recording start.
  - [x] **SignalR replay** &mdash; mock speaks the JSON hub protocol end-to-end: handshake (`{"protocol":"json","version":1}` → `{}`), U+001E-terminated frames, Invocation / StreamItem / Completion / StreamInvocation / CancelInvocation / Ping / Close message types. Server-initiated broadcasts emit from `ReceivedMessages` with per-frame `timestampMs` pacing. Client-initiated invocations pair against recorded `SentMessages` by target method, and the matched Completion's `invocationId` is rewritten to the client's chosen id (same pattern as graphql-transport-ws id-rewriting). Unpaired targets get a synthesised error Completion so clients don't hang. Socket stays open until the client closes (up to a 30s idle guard) so request-response flows can complete after the broadcast is done. Socket.IO replay stays in the deferred queue below.
  - [x] **Socket.IO replay** &mdash; `ReplaySocketIoAsync` speaks the engine.io-on-WebSocket layer end-to-end: accepts the upgrade on `/socket.io/?EIO=4&transport=websocket`, sends the engine.io OPEN packet (`0{sid,pingInterval,pingTimeout,...}`) with a synthetic sid + empty `upgrades` list so clients don't try to negotiate an alternate transport, parses the Socket.IO CONNECT packet to pick up the namespace (default `/` or `/admin,...`), replies with the matching CONNECT ack (`40{sid}` or `40/ns,{sid}`), then emits the recorded events as `42[...]` EVENT frames. The capture-side envelope (`{event,data,timestamp}` — what `BowireSocketIoProtocol.InvokeStreamAsync` persists) is unwrapped into the Socket.IO wire shape; frames already in raw `42[...]` form pass through unchanged. Engine.io PING (`2`) is answered with PONG (`3`) in a background drain loop so long-running replays don't get culled as idle. `ExactMatcher` pairs Socket.IO steps against any GET upgrade on `/socket.io/*` regardless of `httpPath` (the plugin never captures one).
    - **Ack callbacks:** frames whose envelope event name is the reserved marker `__ack__` siphon off the broadcast stream into a FIFO ack-response queue at load time. When the client sends `42N[...]` with an ack id `N > 0` (incl. namespaced form `42/ns,N[...]`), the drain loop pops the next queued response and replies with `43N[<payload-array>]` so the client's callback fires. Order-based pairing — the N-th recorded ack feeds the N-th incoming ack-carrying event. Scalar / object payloads get wrapped in a single-element array so client libraries' destructuring callbacks receive positional args.
    - **Binary events:** frames with a `binary: "<base64>"` envelope field opt into engine.io type-5 BINARY_EVENT. The mock emits the placeholder header (`51-["event",{"_placeholder":true,"num":0}]`) as a text frame, then the decoded attachment bytes as a raw WebSocket binary frame. Optional `data` alongside `binary` rides along as an extra array element. Single-attachment only for now; multi-attachment + BINARY_ACK can ship as a follow-on slice.
  - [x] **SSE `Last-Event-ID` resume** &mdash; each emitted frame now carries `id: <index>` alongside its `data:` line, and the SSE replayer reads `Last-Event-ID` on incoming requests. Reconnecting clients skip every frame whose `Index` is at or below the header value and pick up from the next one. Non-numeric / empty header values replay from the start; values past the end emit nothing. Frame ids come from the existing monotonic `BowireRecordingFrame.Index` — no recorder-side changes needed, old recordings resume just as cleanly as fresh ones.
- [x] **Proactive + timed emission (MQTT)** &mdash; **Phase 2f, shipped.** When a recording carries MQTT publishes the mock spins up an embedded MQTTnet 5.1 broker alongside the HTTP server (opt-in via `MockServerOptions.EnableMqtt`, port picked via `MqttPort` or OS-assigned when `0`). A background scheduler replays each publish at its `capturedAt` offset via `MqttServer.InjectApplicationMessage`, paced by `ReplaySpeed`. QoS, retain, and payload come from the recording; `${uuid}` / `${now}` / ... substitution runs on the payload before emit. 250 ms startup grace so subscribers that are still completing CONNECT+SUBSCRIBE don't miss the opening burst. DIS multicast broadcasts remain on the follow-on list.
  - [x] **Looped replay** &mdash; `MockOptions.Loop` / `--loop` / `Bowire:Mock:Loop` wraps the MQTT proactive emitter's schedule in a do-while that resets the wall-clock origin on each iteration so pacing stays correct across replays. Default stays one-shot. Other replayers (SSE, gRPC streaming, WebSocket, SignalR) are one-shot by construction — the concept only maps onto the proactive-emission protocols.
  - [x] **Plugin-contributed mock emitters** &mdash; `IBowireMockEmitter` (in `Kuestenlogik.Bowire.Mock.Emit`) lets external protocol plugins plug broadcast-style replay into the mock server without adding code to the Bowire core. Emitters advertise an id, claim the recording via `CanEmit`, own their own transport (UDP multicast for DIS, DDS middleware, raw WebSockets, ...), and get started after Kestrel comes up. `MockServerOptions.Emitters` carries the list; the `bowire mock` CLI populates it via `PluginManager.EnumeratePluginServices<IBowireMockEmitter>()`, which scans every loaded plugin ALC. Emitters that throw on start are logged and skipped so a broken plugin can't take the HTTP replayer down. Disposal flows through `MockServer.DisposeAsync` in reverse startup order.
  - [x] **DIS PDU broadcast replay** &mdash; shipped in `Bowire.Protocol.Dis` (`DisMockEmitter`). Opens a UDP socket, joins the configured multicast group (default `239.1.2.3:3000`, overridable via per-step metadata `multicast-group` / `port` / `ttl`), and broadcasts each step's `responseBinary` paced by `capturedAt`. Loop mode + `ReplaySpeed` both honoured, mirroring the MQTT proactive emitter. Round-trip covered by `DisMockEmitterTests.EmitsPduBytesOnMulticastGroup`. Sibling UDP emitter (`Bowire.Protocol.Udp.UdpMockEmitter`) rebroadcasts captured UDP datagrams on the same extension point — also shipped, not originally rostered here. `PluginManager.Inspect` now lists every `IBowireMockEmitter` contribution alongside the existing `IBowireProtocol` / `IBowireProtocolServices` lines, so `plugin inspect` is the single source of truth for "what does this plugin give the host." The `smoke-dis-udp.{sh,ps1}` script runs the full pack → install → inspect chain and greps the output for `DisMockEmitter`, so a regression in the plugin-ALC walker (the path that feeds `bowire mock`'s proactive replay) fails CI loudly. End-to-end verified on 2026-04-24 — smoke exit 0 with both emitters visible.
- [x] **Dynamic values in response bodies** &mdash; **Phase 2b, shipped.** `ResponseBodySubstitutor` resolves `${uuid}`, `${now}`, `${nowMs}`, `${now+N}`, `${now-N}`, `${timestamp}`, and `${random}` per request on REST replay. Unknown tokens pass through verbatim so literal `${...}`-shaped content in a recorded body isn't clobbered. gRPC skipped — binary protobuf and text substitution don't mix; schema-aware substitution parked for later phases. Streamed-frame substitution (Phase 2c onwards) will plug in at the same point when streaming replay lands.
  - [x] **Request-based templating** &mdash; recorded responses can echo values from the live request via `${request.*}` tokens: `${request.method}`, `${request.path}`, `${request.path.N}` (0-based segment) / `${request.path.NAME}` (named path-template binding from the matched `{name}` placeholder), `${request.query.NAME}`, `${request.header.NAME}` (case-insensitive), `${request.body}` (raw), `${request.body.a.b}` (JSON path navigation — dotted keys into objects, integer indices into arrays). `MockHandler` buffers the body once, `ExactMatcher.ExtractTemplateBindings` re-runs the matched template's regex to surface named captures, and the new `RequestTemplate` snapshot exposes all of it to the substitutor. Wired into REST unary and SSE streaming replay; gRPC skipped (binary wire format) and WebSocket/SignalR skipped (no post-upgrade body to template against). Lets a single recording power many test cases — echo endpoints, correlation-id mirrors, parameter echoes — without per-test edits.
- [x] **Path-based matching for REST** &mdash; **Phase 2a, shipped.** `ExactMatcher` now detects OpenAPI-style templates in a step's `httpPath` (any occurrence of `{name}`) and compiles them to cached regexes: each placeholder binds to one path segment (`[^/]+`), literal segments are Regex-escaped so `.` in `/files/{name}.txt` stays a literal dot. Literal-only paths keep the fast string-equality path, preserving Phase-1a behaviour. Binding extraction / dynamic-response substitution is separate and ships in Phase 2b.
  - MockServerOptions now accepts `Port = 0` (OS-assigned), with the bound port exposed as `MockServer.Port` — integration tests no longer rely on a random-range port lottery.
- [x] **Topic-based matching (MQTT)** &mdash; two halves, both shipped:
    - **Proactive**: the MQTT proactive emitter runs `${uuid}` / `${now}` / ... substitution on the publish topic so recorded templates like `sensors/${uuid}/temp` fire on a concrete topic every replay. MQTT wildcards (`+`, `#`) on subscriber filters go through the broker natively.
    - **Reactive**: new `MqttReactiveResponder` hooks the broker's `InterceptingPublishAsync` event. Client publishes on a topic matching a recorded `Duplex`/`ClientStreaming` MQTT step's pattern trigger a paired response. `MqttTopicMatcher` handles the wire wildcards (`+` single-segment, `#` multi-segment trailing) and captures bindings as `${topic.0}`, `${topic.1}`, `${topic.rest}`, which the `ResponseBodySubstitutor` can weave into the response topic and body. MQTT v5 `ResponseTopic` on the incoming publish wins over the recorded template; `CorrelationData` is echoed back so clients can pair request/response. Socket.IO wildcard matching stays deferred with its own replay slice.
- [x] **Cross-protocol coverage — OData + MCP** &mdash; **Phase 2g, shipped.** Replayer dispatch refactored to pick a strategy from the step's *wire-level* characteristics (httpPath + httpVerb → REST/SSE path) rather than the protocol label. That means OData, MCP streamable-HTTP unary, and MCP SSE notifications all replay through the existing REST + SSE paths without protocol-specific code — verified by three integration tests: `$metadata`-shaped OData body, `tools/call` MCP JSON-RPC call, and `notifications/progress` SSE stream.
  - [x] **GraphQL subscription replay (`graphql-transport-ws`)** &mdash; **Phase 2h, shipped.** The mock accepts WebSocket upgrades on the recorded graphql endpoint with the `graphql-transport-ws` subprotocol, speaks the handshake (`connection_init` → `connection_ack`), reads the client's `subscribe` frame to capture its chosen id, and replays the recorded `next`/`complete`/`error` frames with that id rewritten on every outgoing frame. Frames come from `receivedMessages` and can be either raw protocol frames or the `{type:"text", text: <inner>}` channel envelope that the recorder stores; both shapes are unwrapped the same way. If the recording stops before a `complete` is captured, the replayer synthesises one so the client-side iterator terminates. Dispatch triggers on `protocol == "graphql"` + duplex/server-streaming + WS upgrade, so pure WebSocket recordings keep falling through to the Phase-2e path unchanged.

#### Phase 3 — Stateful, chaos, schema-only, capture

Where Bowire can actually differentiate — WireMock has some of this but with a complex DSL; Postman mocks are stateless and cloud-only.

- [x] **Stateful mode** &mdash; **Phase 3b, shipped.** `MockOptions.Stateful` (CLI: `--stateful`, `--stateful-once`) flips the matcher into strict-order mode: only the step at the current cursor is eligible to match, every successful hit advances the cursor, and out-of-order requests miss cleanly. `StatefulWrapAround` (default `true`, or `--stateful-once` to flip it off) controls end-of-recording behaviour — wrap back to step 0 for continuous loops, or start returning misses once the scripted flow is exhausted. The cursor resets on `RecordingWatcher` reloads so editing the file mid-test restarts the sequence. Composes with chaos injection: both knobs apply to whichever step the cursor picks. Lock-serialised read-check-advance so concurrent requests serialise on ordering without blocking replay I/O.
- [x] **Runtime scenario switching** &mdash; `POST /__bowire/mock/scenario {"path":"..."}` hot-swaps the active recording in-flight; `GET /__bowire/mock/status` reports the current one. Gated by `MockOptions.ControlToken` / `--control-token` / `Bowire:Mock:ControlToken`: unset → every request to `/__bowire/mock/*` returns 404 (surface not advertised); set → `X-Bowire-Mock-Token` header must match, mismatches return 401. Path-traversal defence constrains swap targets to the initial recording's directory subtree (absolute paths + `..` escapes rejected). CI pipelines toggle between happy-path and error-path scenarios without restarting the mock.
- [x] **Chaos injection** &mdash; **Phase 3a, shipped.** `MockOptions.Chaos` + `bowire mock --chaos "latency:100-500,fail-rate:0.05"` (parsed via `ChaosOptions.Parse`) inject a per-request delay drawn from the latency range and short-circuit a configurable percentage of matched requests with a fail-status body (default `503`, overridable via `fail-status:<code>`). Chaos applies only on a matched step — unmatched paths still surface as 404 / pass-through so high fail rates don't mask recording-coverage gaps. Latency fires before dispatch, so the first byte of a streaming response carries the chaotic delay while the captured frame cadence (or `ReplaySpeed`) stays untouched. Supported key combinations: `latency:<ms>`, `latency:<min>-<max>`, `fail-rate:<0..1>`, `fail-status:<code>`.
- [x] **Schema-only mock (OpenAPI)** &mdash; **Phase 3d, shipped.** `MockServerOptions.SchemaPath` / `bowire mock --schema <path>` loads an OpenAPI 3 document (JSON or YAML — SharpYaml reader registered explicitly) and synthesises a recording at startup via `SchemaRecordingBuilder`: every operation becomes one step whose response body is generated by `OpenApiSampleGenerator` from the declared response schema. Type-aware generation (integer → `1` or `minimum`, number → `1.5`, string → `"sample"` or a format-specific placeholder for `date-time`/`uuid`/`email`/`uri`/`byte`/..., boolean → `true`, array → 3 items, object → recursive, enum → first value); `example`/`default` on the schema always wins. Recursion capped at 8 so cyclic `$ref`s terminate. Composes with chaos/stateful/miss-capture — the synthesised recording is just a recording as far as the middleware is concerned. `--recording` and `--schema` are mutually exclusive.
- [x] **Schema-only mock (gRPC)** &mdash; `MockServerOptions.GrpcSchemaPath` / `bowire mock --grpc-schema <path.pb>` loads a protobuf `FileDescriptorSet` binary (produce with `protoc --descriptor_set_out=path.pb --include_imports your.proto`) and synthesises a gRPC recording whose every service method gets a binary sample response from a hand-rolled `ProtobufSampleEncoder` (walks the `MessageDescriptor`, writes wire-format bytes directly via `CodedOutputStream` since Google.Protobuf C# has no `DynamicMessage`). The synthesised steps carry a base64 `schemaDescriptor` of the full FileDescriptorSet so the existing `DescriptorPool` + `Grpc.Reflection.ReflectionServiceImpl` path lights up automatically — clients hitting the mock auto-discover the services. Server-streaming methods emit three sample frames. Reuses the existing `UnaryReplayer` gRPC branches; the matcher-first reorder in `MockHandler` lets the mock shadow grpc-dotnet's catch-all `Unimplemented` fallback for paths the recording actually covers. Mutex with `--recording` / `--schema`.
- [x] **Schema-only mock (GraphQL)** &mdash; `MockServerOptions.GraphQlSchemaPath` / `bowire mock --graphql-schema <path.graphql>` loads an SDL file via `GraphQL-Parser` and answers every `POST /graphql` request by parsing the incoming query, walking the schema for each selected field, and returning a sample-valued JSON payload shaped to match the selection set. Unlike the OpenAPI / gRPC modes the response is synthesised per request (not pre-baked into a recording) because GraphQL responses are selection-set-dependent: `{ user { name } }` and `{ user { name, email } }` against the same schema must yield differently-shaped bodies. Scope: Query operations only (Mutations / Subscriptions return an explicit error); scalars, enums (first value), lists (3 items), nullable wrappers, aliases, and `__typename` all honoured; interfaces / unions return `null` since the mock can't pick a concrete implementor without runtime resolvers; recursion capped at depth 8. Custom `schema { query: Root }` blocks rename the root type correctly. Common custom scalars (`DateTime`, `UUID`, `Email`, `URL`, `JSON`) get type-appropriate placeholders. Mutex with `--recording` / `--schema` / `--grpc-schema`.
- [x] **Request capture on miss** &mdash; **Phase 3c, shipped.** `MockOptions.CaptureMissPath` / `bowire mock --capture-miss <path>` persists every unmatched request to the named file as a placeholder `BowireRecordingStep` (verb, path, headers as metadata, body up to 1 MiB with a truncation marker above). Response/status are blank — the user fills them in and replays. Appends to an existing single-recording document or creates a fresh one on first miss; hop-by-hop headers (`:authority`, `Connection`, `Transfer-Encoding`, …) are filtered so the captured steps describe the *semantic* request, not the wire connection. Spec-authoring in reverse: exercise the mock, collect what wasn't covered, merge back into the main recording.
  - [x] **gRPC miss-capture** &mdash; same file format and CLI knob as REST. The HTTP/2 `:path` is split into `service` + `method` (matching `ExactMatcher.MatchesGrpcPath`), the first length-prefixed request frame is base64-encoded into the new `requestBinary` field (5-byte envelope stripped so the user sees the protobuf payload, not the wire framing), and `responseBinary` is left blank for the user to fill in. Malformed or short bodies still produce a step so the user knows the RPC path was hit — they just get a null `requestBinary`. Multi-message client-streaming misses keep only the opener, which is usually enough to identify the RPC; the user can upgrade `methodType` from the default `Unary` if needed.

#### Not in scope / deliberately deferred

- **DIS replay** — the simulation/DIS plugin isn't shipped yet; once it is, recording → mock for DIS becomes a trivial extension of Phase 2's "remaining protocols" track. Kept on the radar but not part of the three core phases.
- **GUI mock builder** &mdash; **shipped** on top of the Freeform Request Builder: the freeform pane grew a collapsible "Mock Response" section (status + body textarea), a "Save as Mock Step" button that freezes the current request + response into the active recording (spinning up a "Manual mocks" recording if none is active), and the Execute path now captures into the active recording too — same `captureRecordingStep` contract as discovery-first invokes, so the existing `bowire mock --recording <file>` path replays GUI-built mocks without special-casing. For REST the method field is parsed as `<VERB> <path>` or bare `/path` and the derived `httpPath`/`httpVerb` land on the recording step, so the Phase-1a wire matcher keys on them. An Execute auto-fills the Mock Response textarea from the real response on first hit, so "capture once, tweak, save as mock" is two clicks.

### HAR Import — Playwright + DevTools as recording sources

Bowire already exports HAR (Phase &lt;v0.9.0&gt;, recording manager → "Export HAR") so the captured multi-protocol traces survive in DevTools, Charles, Insomnia, etc. The other half is missing: ingesting HAR files *into* a Bowire recording. With native HAR import, every tool that produces HAR becomes a recording source — most notably **Playwright** (`browser.newContext({ recordHar: …})`), Chrome / Firefox DevTools ("Save all as HAR"), Charles, Fiddler, Postman, and ETW HTTP traces. The user replays the import as a smoke-test, fronts it with `bowire mock`, or merges it into an existing recording — all paths the recorder already supports for native captures.

The Bowire recorder and Playwright's HAR recorder are conceptually parallel — both wrap the live network layer and persist full request/response pairs with timing — so importing HAR is *symmetric* to recording: same downstream tooling (replay, mock, convert-to-tests, HTML report). The asymmetry is protocol scope: HAR covers HTTP/HTTPS only, while Bowire's native recorder spans gRPC, GraphQL, MQTT, WebSocket, SSE, SignalR, MCP, Socket.IO. HAR import therefore lights up REST and (where the source captured them) HTTP/2-framed gRPC-Web; the streaming protocols stay in the native-recorder lane.

- [x] **`bowire import har <file.har> [--out recording.bwr] [--name "…"]`** — shipped. CLI command parses HAR 1.2, maps `entries[]` to `BowireRecordingStep[]` (verb / path / headers / postData / status / response body / timing all forwarded into the v2 `BowireRecording` shape). By default writes `<har-basename>.bwr` next to the input; `--out -` streams to stdout for piping into `bowire mock`. Numeric / GUID-shaped tail segments stay out of the derived service / method name so the workbench sidebar groups requests by collection rather than by id. 20 unit tests cover the mapping plus the file-IO entry point. UI button + per-entry filter + merge mode stay deferred to the items below.
- [ ] **UI import button** — recording manager toolbar gets "Import HAR" next to the existing "Export HAR" / "Export JSON" buttons. Opens a file picker, runs the same parser client-side (`importHarAsRecording` in `wwwroot/js/recording.js`, sibling of `exportRecordingAsHar`), persists via the existing `persistRecordings` debounced sync. No new server endpoint — the parse is pure client-side.
- [ ] **gRPC-Web detection** — when a HAR entry's request body looks like length-prefixed protobuf (5-byte framing: 1 byte compression flag + 4 bytes big-endian length) and the `Content-Type` is `application/grpc-web` or `application/grpc-web-text`, classify the step as gRPC instead of REST: split the URL path into `service` + `method`, base64-encode the binary frame into `requestBinary` / `responseBinary` (matching the v2 recording-format from Phase 1b of the mock work), so replay and mock both work without re-encoding. Falls back to REST classification if the framing doesn't match.
- [ ] **Per-entry filter** — preview pane in the import dialog lists every HAR entry with verb / path / status / size; checkboxes let the user keep only the calls they care about (typical Playwright HAR includes static assets — `.js`, `.css`, fonts — that pollute a recording). Default: pre-uncheck obvious static-asset MIME types (`text/css`, `image/*`, `font/*`, `text/javascript` when the URL ends with `.js`); user can flip them back on.
- [ ] **Merge mode** — instead of "create new recording", offer "Append to: \<existing recording dropdown\>". Use case: existing Bowire recording captures gRPC + WebSocket, Playwright HAR adds the REST calls the UI fired against the same backend during the test — combined recording covers the whole flow, replays as one mock.
- [ ] **Playwright integration page in docs** — `docs/integrations/playwright.md` walking through the loop: write a Playwright test → record HAR → `bowire import har` → `bowire mock --recording …` → re-run the Playwright test against the local mock (zero-backend CI). Sister page for browser-DevTools captures (`docs/integrations/browser-devtools.md`) covering the same loop manually.
- [ ] **Round-trip test** — golden-file test (`HarImportRoundTripTests`) that takes a fixture HAR generated by Playwright against a known service, imports it, exports it again, and asserts the second HAR matches the first byte-for-byte (modulo `_creator` metadata). Catches mapping drift between import and export when either side evolves.

The mapping is straightforward — HAR is well-specified, the recording format is already designed to survive lossless export — so this is mostly plumbing, no new architectural surface. Natural target: alongside the v0.9.x recordings polish, before the freeform-builder work since import gives users a fast on-ramp without manually crafting requests.

### Freeform Request Builder (Postman-style "New Request")

Currently Bowire is discovery-first: services and methods come from server reflection, OpenAPI, introspection, etc. The freeform request builder flips this — the user creates a request from scratch without a discovered schema.

- [ ] **"New Request" entry point** — button in the sidebar header / command palette / landing page. Opens a blank request pane instead of selecting a discovered method.
- [ ] **Protocol picker** — dropdown or tab row at the top of the request pane: gRPC, REST, GraphQL, SignalR, MQTT, WebSocket, SSE, MCP. Selection determines which fields appear below (URL, topic, query, hub method, etc.) and which plugin handles the invocation.
- [ ] **Dynamic input fields per protocol**:
    - **REST**: HTTP verb + URL + headers + body (JSON/form-data/raw) — closest to Postman's core flow
    - **gRPC**: target URL + fully-qualified method name + JSON body + metadata headers. Optional: paste a `.proto` snippet so Bowire knows the schema without full reflection.
    - **GraphQL**: endpoint URL + query editor + variables editor + headers
    - **MQTT**: broker URL + topic + payload + QoS + retain flag
    - **SignalR**: hub URL + method name + JSON args
    - **WebSocket**: URL + message payload + sub-protocol
    - **SSE**: URL (read-only stream — no request body)
    - **MCP**: server URL + tool/resource/prompt name + JSON input
- [ ] **No schema required** — the request goes through the same `IBowireProtocol.InvokeAsync` / `InvokeStreamAsync` path, but the user provides all parameters manually instead of picking from a discovered list. Plugins need a small extension so they accept ad-hoc invocations without prior discovery.
- [ ] **Save as collection item** — persist the freeform request into a named collection (separate from recordings). Collections are portable JSON files, importable/exportable, shareable via workspace files.
- [ ] **History integration** — freeform requests land in the same call history as discovered-method calls so the user can re-run them from history.
- [ ] **Auto-discover after first call** — once a freeform gRPC/REST/GraphQL request succeeds, offer to run discovery against that URL so the user gets the full schema-backed experience for subsequent calls.

### Bowire.Mcp — self-service MCP server (drive Bowire from AI agents)

Bowire has three different MCP-shaped surfaces and they belong in three different packages so users (and the codebase) don't conflate them:

| Direction | Role | Package |
|---|---|---|
| `Bowire → external MCP server` | Bowire is the **client**, discovering Tools / Resources / Prompts on a foreign MCP host | `Kuestenlogik.Bowire.Protocol.Mcp` (`BowireMcpProtocol`, `McpDiscoveryClient`) |
| `Bowire as MCP adapter for fremd-protocols` | Bowire is the **server**, wrapping discovered gRPC / REST / GraphQL services as MCP tools so AI agents can drive any backend through Bowire | `Kuestenlogik.Bowire.Protocol.Mcp` (`McpAdapterEndpoints`, `McpAdapterServer`) |
| **`Bowire as MCP server for itself`** | **Bowire is the server**, exposing its **own** operations (discover, invoke, record, replay, env, mock) as MCP tools so AI agents drive the workbench itself | **`Kuestenlogik.Bowire.Mcp`** (new — sibling of `Kuestenlogik.Bowire.Mock`) |

The third row is what this roadmap entry is about. Phase 1+2 + most of Phase 3+5 are now shipped on top of the official `ModelContextProtocol` C# SDK (1.2.0); the residual work is in a dedicated phase below.

- [ ] **Workspace files** — a single `.bowire` JSON file at the repo root that captures environments + tests + URL config + auth helpers. Bowire auto-loads it on startup, the file is gitignorable or shareable. Replaces the per-user `~/.bowire/environments.json` for repo-local configs.
- [ ] **OpenAPI schema export** — inverse of the existing OpenAPI import. Generates an OpenAPI 3.1 document from the discovered REST methods (path templates, parameter buckets, body schemas) so users can publish or commit a schema based on what Bowire already knows about a running service.
